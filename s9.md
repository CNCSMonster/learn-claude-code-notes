# s9: Agent Teams - Lead 与队友通信机制

## 核心问题

1. 每个 agent 是否都能够给任何一个 teammate 发送消息？
2. 是否只能够读取自己的 inbox？
3. 队友如何知道其他 teammate 在哪里？

## 答案

### 1. 发送和读取权限

```
每个 agent 可以：
✓ 给任何 teammate 发送消息（只要知道名字）
✓ 只能读取自己的 inbox
```

### 2. 通信工具

**Lead 有 9 个工具**：
```python
TOOLS = [
    {"name": "bash", ...}, {"name": "read_file", ...},
    {"name": "write_file", ...}, {"name": "edit_file", ...},
    {"name": "spawn_teammate", ...}, {"name": "list_teammates", ...},
    {"name": "send_message", ...}, {"name": "read_inbox", ...},
    {"name": "broadcast", ...},
]
```

**队友只有 6 个工具**（相比 Lead 少了 `spawn_teammate`、`list_teammates`、`broadcast`）：
```python
def _teammate_tools(self) -> list:
    return [
        {"name": "bash", ...}, {"name": "read_file", ...},
        {"name": "write_file", ...}, {"name": "edit_file", ...},
        {"name": "send_message", ...}, {"name": "read_inbox", ...},
    ]
```

### 3. 队友如何知道其他成员？

**队友没有 `list_teammates` 工具**，文档中没有明确说明队友如何发现其他成员。

**唯一明确的方式：Lead 在 spawn 的 prompt 中告知**
```python
spawn_teammate(
    name="alice", 
    role="coder", 
    prompt="You are alice, the coder. Bob is the reviewer. "
           "When you finish coding, send bob a message for review."
)
```

**注意**：
- 队友理论上可以用 `read_file` 读 `.team/config.json`，但文档和代码都没有明确说明这是推荐方式
- 队友也可以通过 Lead 或其他队友发送的消息获知其他成员的存在

## 完整通信图

```
┌─────────────────────────────────────────────────────────┐
│                      Lead (领导)                         │
│  工具：spawn_teammate, list_teammates, send_message     │
│        read_inbox, broadcast                            │
└───────────────┬─────────────────────┬───────────────────┘
                │                     │
      send_message         send_message
      to="alice"           to="bob"
                │                     │
                v                     v
┌───────────────────────┐ ┌───────────────────────┐
│    Alice (Coder)      │ │   Bob (Reviewer)      │
│  工具：send_message   │ │  工具：send_message   │
│        read_inbox     │ │        read_inbox     │
│                       │ │                       │
│  - 写代码              │ │  - 审核代码            │
│  - 可通知 Reviewer    │ │  - 可通知 Lead         │

└───────────────────────┘ └───────────────────────┘
                │                     │
                └──────────┬──────────┘
                           │
                 send_message
                 to="lead"
                           ↓
              ┌────────────────────────┐
              │  lead.jsonl (Lead 邮箱) │
              └────────────────────────┘
```

## 典型工作流

```
# 用户给 Lead 下指令
"Spawn alice as coder and bob as reviewer. Have them work together."

# Lead 的操作
1. spawn_teammate(name="alice", role="coder", 
   prompt="You are alice, the coder. Bob is the reviewer. 
           When done, send bob a message for review.")

2. spawn_teammate(name="bob", role="reviewer",
   prompt="You are bob, the reviewer. Alice will send you code.")

3. send_message(to="alice", content="Start working on auth.py")

# 后续通信（队友之间直接进行）
Alice → send_message(to="bob", content="Please review auth.py")
Bob   → send_message(to="lead", content="Review complete. 2 issues.")

# Lead 查看结果
Lead → read_inbox()  # 收到 Bob 的汇报
```

## 关键设计点

| 特性 | 设计 |
|------|------|
| 服务发现 | 无自动发现，靠 Lead 协调告知 |
| 通信媒介 | `.team/inbox/{name}.jsonl` 文件 |
| 消息持久化 | append-only JSONL，读完清空 |
| Lead 角色 | 信息中心 + 协调者 |
| 队友自主性 | 有限，依赖 Lead 提供上下文 |

## JSONL

**JSONL** = 每行是一个完整 JSON。

```
{"type": "message", "from": "alice", "content": "hello"}
{"type": "message", "from": "bob", "content": "hi"}
```

**为什么选它**：追加写入无需读改写整个文件。

**读写模式**：尾部追加，顺序读取，读完清空。

## 潜在竞争条件

**问题**：读写无锁保护，可能丢失消息。

```
时间线：
T0: Lead 写入 line3     → 文件："line1\nline2\nline3\n"
T1: Alice 读取全部      → 读到 ["line1", "line2", "line3"]
T2: Bob  写入 line4     → 文件："line1\nline2\nline3\nline4\n"
T3: Alice 清空文件      → 文件：""  ← line4 丢失！
```

**原因**：`read_inbox()` 的"读"和"清空"之间有间隙，非原子操作；代码未加锁。

**修复思路**：每个 inbox 文件一把锁，保护读写操作。

## 总结

- **Lead 是信息中心**：负责创建队友、告知团队成员关系、协调工作
- **队友没有自动发现机制**：简化设计，但增加 Lead 负担
- **通信是对称的**：任何 agent 都可以给任何其他 agent 发消息
- **读取是隔离的**：每个 agent 只能读自己的 inbox
