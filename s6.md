## 补充理解：Message 结构与工具调用

### `isinstance` 的作用

在 `micro_compact` 中，`isinstance` 用于类型检查，确保代码安全地处理消息结构：

```python
# 实际代码 (s06_context_compact.py)
def micro_compact(messages: list) -> list:
    tool_results = []
    for msg_idx, msg in enumerate(messages):
        if msg["role"] == "user" and isinstance(msg.get("content"), list):
            for part_idx, part in enumerate(msg["content"]):
                if isinstance(part, dict) and part.get("type") == "tool_result":
                    tool_results.append((msg_idx, part_idx, part))
    # ...
```

- 第一个 `isinstance(msg.get("content"), list)`: 检查 `content` 是否是列表（因为 `content` 也可能是字符串）
- 第二个 `isinstance(part, dict)`: 检查 `part` 是否是字典（列表中的元素可能有不同类型）

这是**防御性编程**，避免在非预期类型上操作导致崩溃。

### Message 内容结构

**每个 `part` 是一个 content block**，可能是：
- `tool_result` - 工具调用结果
- `text` - 纯文本内容
- 其他类型的 block

**什么时候 `content` 是列表？**

当一次工具调用产生多个结果或需要组合多种内容时，API 使用列表格式：

```python
# 场景 1: 单个工具调用结果
{"role": "user", "content": [{"type": "tool_result", "tool_use_id": "xxx", "content": "..."}]}

# 场景 2: 多个工具调用结果（打包成一条 message）
{
    "role": "user",
    "content": [
        {"type": "tool_result", "tool_use_id": "toolu_1", "content": "read_file 的结果"},
        {"type": "tool_result", "tool_use_id": "toolu_2", "content": "bash 命令的结果"},
    ]
}

# 场景 3: 普通文本消息（content 是字符串，不是列表）
{"role": "user", "content": "请帮我读取这个文件"}
```

### 工具调用与回复的对应关系

```
assistant: 一条 message 可以包含多个 tool_use
{
    "role": "assistant",
    "content": [tool_use_1, tool_use_2, tool_use_3]
}
           ↓
user: 所有结果打包成一条 message
{
    "role": "user",
    "content": [tool_result_1, tool_result_2, tool_result_3]  # 批量返回
}
```

这也是为什么 `micro_compact` 需要遍历 `content` 列表中的每个 `part` 来找 `tool_result` —— 因为一条 user message 里可能藏着多个工具的结果。
